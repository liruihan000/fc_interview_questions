# 问题5：从0到1落地AI产品

> 如果让你在公司内部落地一个AI产品，你会如何从0到1推进？

## 推进路线图

```
Phase 1 (Week 1-2)  需求调研 & 场景筛选
Phase 2 (Week 3-4)  PoC验证 & 技术选型
Phase 3 (Week 5-8)  MVP开发 & 内部测试
Phase 4 (Week 9-12) 灰度发布 & 数据驱动迭代
Phase 5 (Week 13+)  全量推广 & 持续优化
```

## 各阶段关键动作

**Phase 1：需求调研**
- 业务访谈识别痛点——核心问题是"什么最痛"，而非"什么能加AI"
- 量化现状：耗时、成本、错误率，为后续衡量ROI建立基线
- 优先选择可嵌入用户现有工作流的场景（Copilot、Notion AI的成功均源于此；Humane AI Pin试图创造新交互范式，最终失败）
- 同步明确商业模式：附加订阅 / 按量计费 / 捆绑销售

**Phase 2：PoC验证**
- 最简方案验证（直接调API + Prompt，不过早工程化）
- 定义**成功标准**（准确率>90%、处理时间缩短50%等）
- 定义**Kill Criteria**——明确放弃条件，避免沉没成本（Google Bard因竞争恐惧跳过充分验证仓促发布，当天市值蒸发$1000亿）
- 技术选型：模型选择、RAG vs Fine-tuning、架构方案

**Phase 3：MVP开发**
- 最小功能集，解决核心问题；建立Golden Dataset和基础监控
- **场景聚焦优于功能堆叠**：Cursor只做三件套（Tab补全、Cmd+K、Chat），Kimi聚焦"超长上下文"——先建立用户心智再扩展
- 半年窗口期：2026行业共识是Agent产品需在半年内快速验证价值

**Phase 4：灰度发布**
- 5%用户灰度，A/B测试AI方案 vs 传统方案
- 根据数据调整Prompt、模型、流程
- 渐进式发布是业界标准做法：Notion AI等待列表制、GitHub Copilot邀请制内测均采用此策略

**Phase 5：全量推广**
- 建立运维SOP，成本优化（模型路由、缓存），持续监控质量指标
- 构建**数据飞轮**：使用越多 → 数据越多 → 模型越好 → 体验越好 → 使用越多

## 组织层面

- **选对试点团队**：找愿意尝新、能容错、反馈质量高的团队
- **快速展示价值**：2周内出Demo，让stakeholder看到可能性
- **设立AI Champion**：各部门设推广人
- **建立信任**：透明展示AI局限性，不过度承诺

## 业界案例速览

| 类型 | 案例 | 核心策略 |
|------|------|---------|
| 嵌入式AI | **GitHub Copilot** | 嵌入IDE已有工作流，从补全→Chat→Workspace渐进扩展 |
| 产品附加AI | **Notion AI** | 嵌入高粘性产品，场景聚焦写作辅助，附加订阅降低决策门槛 |
| AI-Native | **Cursor** | Fork VS Code做独立产品，每周发版，社区口碑驱动增长，ARR破$1亿 |
| 中国C端 | **豆包（字节）** | C端App + B端API双线并行，极低定价抢占生态 |
| 中国开源 | **通义千问（阿里）** | 开源建社区，钉钉+阿里云切企业场景 |
| 差异化突围 | **Kimi（月之暗面）** | 单一卖点"超长上下文"精准定位知识工作者 |

**失败教训：**
- **Google Bard**：竞争恐惧驱动仓促发布 → PoC不可省略
- **Humane AI Pin**：创造新交互但体验不及格 → 必须嵌入已有工作流
- **Amazon Alexa AI转型**：无清晰商业模式 → 技术升级不能替代PMF

## 核心方法论

1. **嵌入已有工作流 > 创造新工作流**
2. **场景聚焦 > 功能堆叠**
3. **渐进式发布 > 一步到位**
4. **商业模式前置 > 先免费再想变现**
5. **数据飞轮驱动长期壁垒**
