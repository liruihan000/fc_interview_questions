# 问题3：如何判断业务场景是否适合用大模型？

> **原题**：公司希望"所有流程都加AI"。你如何判断某个业务场景是否真的适合用大模型？

## 分析传统方式的可行性

可以首先判断传统技术能否解决 看规则引擎、传统ML能否解决。所有场景替换agent可能会导致错误率提高和token成本的问题。

业界方案：可以围绕这5个维度判断，不一定每个场景都要跑一遍打分，但思考过程相同：

## 五个判断维度

| 维度 | 不适合 | 可考虑 | 强适合 |
|------|--------|--------|--------|
| **容错性** | 零容错（金融结算） | 可人工复核（合同审查） | 允许偏差（摘要、建议） |
| **数据特征** | 纯数值/结构化 | 半结构化 | 非结构化文本、多模态 |
| **ROI** | 低频、人工成本低 | 中频、有一定人力开销 | 高频、重复、人力密集 |
| **可验证性** | 输出难以判断对错 | 需专业人员验证 | 可快速自动化验证 |
| **现有方案差距** | 规则已解决90%+ | 传统ML能解决60-80% | 传统方案无法解决 |

**判定标准**：总分≥18 → 强烈推荐；13-17 → 值得PoC验证；≤12 → 不建议使用大模型。

---

## 决策流程

```
收到"加AI"需求
│
├── 能用规则/传统ML解决？ → YES → 不用大模型
│
└── NO → 5维度打分
         │
         ├── ≤12分 → 不建议，给出替代方案
         │
         ├── 13-17分 → PoC验证（2周内出结论，定义量化成功标准）
         │
         └── ≥18分 → 推进，选技术路线（→ RAG / Fine-tuning / Agent）
```

## 用FC的场景举例

**SEC Filing中国背景公司筛选：适合**

容错性OK（筛选阶段可以有误判，后面有人工复核），数据是非结构化SEC文档需要语义理解，每天5000条人工筛不现实，而且关键词匹配搞不定VIE结构这种需要推理的case。各个维度都指向"该用LLM"。

**每月固定格式财务对账：不适合**

纯结构化数值，零容错，月频，SQL脚本完全解决。AI可能提高错误率。

## 容易误判的场景

- "用LLM生成合规报告" — 金融合规文件零容错，LLM幻觉风险不可接受，只能辅助人工审核
- "用LLM做数据清洗" — 结构化数据用Pandas/SQL更快更准更便宜

也有反过来的：
- "内部知识库太乱没法用" — RAG可以从非结构化文档中检索，并非无法使用。
- "数据量太小" — RAG不需要训练数据，有文档就能用或者现在经过验证的数状渐进披露文件查询的方式就能解解决大部分问题。

## comments

1. **先算ROI**：判断是否值得
2. **PoC要定量化标准**：比如"准确率>90%"、"处理时间缩短50%"，不是"感觉还行"
3. **"All-in AI"可能暂时是陷阱**：识别高价值、高频、容错的场景优先落地，但长期来看是未来
4. **确认适合后的下一步**：技术路线选型见 [Q6 RAG vs Fine-tuning & Multi-Agent](./q6_rag_finetuning_multiagent.md)
