# 问题3：如何判断业务场景是否适合用大模型？

> **原题**：公司希望"所有流程都加AI"。你如何判断某个业务场景是否真的适合用大模型？

---

## 核心立场

**不是所有问题都需要大模型。** 先问"不用AI能不能做"——如果规则引擎、传统ML能解决，不要用大模型。ROI是第一性原则。

---

## 评估框架：5维度打分

| 维度 | 1分（不适合） | 3分（可考虑） | 5分（强适合） |
|------|-------------|-------------|-------------|
| **容错性** | 零容错（金融交易结算） | 可人工复核（合同审查） | 允许偏差（摘要、建议） |
| **数据特征** | 纯数值/结构化 | 半结构化 | 非结构化文本、多模态 |
| **ROI** | 低频、人工成本低 | 中频、有一定人力开销 | 高频、重复、人力密集 |
| **可验证性** | 输出难以判断对错 | 需专业人员验证 | 可快速自动化验证 |
| **现有方案差距** | 规则已解决90%+ | 传统ML能解决60-80% | 传统方案无法解决 |

**判定标准**：总分≥18 → 强烈推荐；13-17 → 值得PoC验证；≤12 → 不建议使用大模型。

---

## 决策流程

```
收到"加AI"需求
│
├── 能用规则/传统ML解决？ → YES → 不用大模型
│
└── NO → 5维度打分
         │
         ├── ≤12分 → 不建议，给出替代方案
         │
         ├── 13-17分 → PoC验证（2周内出结论，定义量化成功标准）
         │
         └── ≥18分 → 推进，选技术路线（→ RAG / Fine-tuning / Agent）
```

## 实战案例：用框架评估FC业务场景

### 案例1：SEC Filing中国背景公司筛选 → **适合（21分）**

| 维度 | 得分 | 理由 |
|------|------|------|
| 容错性 | 4 | 筛选阶段允许误判，后有人工复核 |
| 数据特征 | 5 | 非结构化SEC文档，需语义理解 |
| ROI | 5 | 5000条/天，人工筛选不现实 |
| 可验证性 | 4 | 可通过抽样对比验证准确率 |
| 现有方案差距 | 3 | 关键词匹配能覆盖部分，但VIE结构等需要推理 |

### 案例2：每月固定格式财务对账 → **不适合（8分）**

| 维度 | 得分 | 理由 |
|------|------|------|
| 容错性 | 1 | 财务数据零容错 |
| 数据特征 | 1 | 纯结构化数值 |
| ROI | 2 | 月频，规则脚本即可 |
| 可验证性 | 3 | 可自动对账验证 |
| 现有方案差距 | 1 | SQL + 脚本完全解决 |

## 常见误判场景

| 看似适合，实际不适合 | 原因 |
|---------------------|------|
| "客服全部换AI" | 投诉/赔付场景容错性极低，需人工兜底 |
| "用LLM做数据清洗" | 结构化数据用Pandas/SQL更快更准更便宜 |
| "AI审批所有合同" | 法律责任不可转移给AI，只能辅助不能替代 |

| 看似不适合，实际适合 | 原因 |
|---------------------|------|
| "内部知识库太乱没法用" | RAG恰好擅长从非结构化文档中检索，乱≠不能用 |
| "我们数据量太小" | RAG不需要训练数据，有文档就能用 |

---

## 实操要点

1. **ROI先算账**：企业关注点已从"技术先进"转向"可衡量的业务价值"
2. **PoC要定量化成功标准**：如"准确率>90%"、"处理时间缩短50%"，不是"感觉还行"
3. **"All-in AI"是陷阱**：识别高价值、高频、容错的场景优先落地
4. **Agent的价值密度**：只有解决高价值问题的Agent才能存活，简单任务用单次Prompt调用
5. **确认适合后的下一步**：技术路线选型见 [Q6 RAG vs Fine-tuning & Multi-Agent](./q6_rag_finetuning_multiagent.md)
